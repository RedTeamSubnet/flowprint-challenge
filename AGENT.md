# AGENT Guide

This repository implements the OS Classification challenge. Use this
file as the default playbook for coding agents working in the repository.

## Challenge Contract

`miner_output.commit_files` contains exactly:

- `train.py`
    - invoked as `python train.py <training_csv> <model_json>`
    - must train from the CSV in `sys.argv[1]`
    - must write valid JSON to `sys.argv[2]`
- `submissions.py`
    - must expose `detect_os(features, model) -> str`
    - receives one feature row and the parsed training JSON

The goal is multiclass operating-system classification. Valid labels are:

- `Android`
- `iOS`
- `Windows`
- `Linux`
- `Chromium OS`
- `Mac OS`

The final score is macro F1 across the OS classes.

## Mandatory Dataset

The source dataset is:

- `volumes/storage/flowradar-challenge/data/flow_data_sampled_350k.csv`

Despite the `.csv` suffix, this file is Parquet. Convert it into real CSV
train/test files before running the platform scorer. The target column is
`device_os`; all other columns are inference features.

Recommended generated paths:

- training: `volumes/storage/flowradar-challenge/data/os_train_data.csv`
- scoring: `volumes/storage/flowradar-challenge/data/os_test_data.csv`

Environment variables:

- `FLR_CHALLENGE_TRAIN_CSV_PATH="{data_dir}/os_train_data.csv"`
- `FLR_CHALLENGE_TEST_CSV_PATH="{data_dir}/os_test_data.csv"`

## Execution Flow

1. `/score` receives `commit_files` containing `train.py` and `submissions.py`.
2. The challenge starts an isolated FlowRadar container.
3. Both scripts and the generated OS training CSV are mounted read-only.
4. The challenge calls the container's `POST /train`.
5. The container runs `train.py` and loads its temporary model JSON.
6. The challenge replays the generated OS test CSV.
7. Each row is sent as `{"products": features}` after removing `device_os`.
8. `detect_os(features, model)` returns an OS label string.
9. The scorer computes the macro F1 score.
10. The container and temporary model are removed after scoring.

Miner code must never execute in the challenge API process.

## Data Handling

- Empty CSV cells are sent to inference as JSON `null`.
- JA4 and sequence columns may be strings.
- Numeric values may be Python `int` or `float`.
- Inference must tolerate missing or null optional fields.
- Training must read `device_os` as the label.
- Avoid unsafe casts, division by zero, and assumptions that every feature is
  present.

## Constraints

- Training must finish within `FLR_CHALLENGE_TRAINING_TIMEOUT_SECONDS`.
- Model JSON must remain below `FLR_CHALLENGE_MODEL_JSON_SIZE_LIMIT`.
- Both scripts must respect the configured submission line limit.
- Request misses above `FLR_CHALLENGE_ACCEPTABLE_MISS_COUNT` stop replay early.
- Inference must return a real OS label string.
- Do not embed pretrained, serialized, encoded, or hard-coded learned weights
  in `train.py` or `submissions.py`.
- All learned weights must be generated by `train.py` from the mandatory OS
  training dataset during the current scoring run.
- `submissions.py` may only consume generated weights through its `model`
  argument and must not include fallback embedded model state.
- Do not use bypass, obfuscation, environment escape, or host-access techniques.

## Local Workflow

1. Convert `flow_data_sampled_350k.csv` from Parquet into train/test CSV files.
2. Configure `.env` with the OS train/test CSV paths.
3. Run `./skills/challenge-setup/scripts/setup.sh --build`.
4. Run `./skills/challenge-setup/scripts/healthcheck.sh`.
5. Run `python3 skills/challenge-score/scripts/check_score.py`.
6. Inspect `/telemetry`, `/results`, and challenge logs.
7. Iterate on both training and inference behavior.
